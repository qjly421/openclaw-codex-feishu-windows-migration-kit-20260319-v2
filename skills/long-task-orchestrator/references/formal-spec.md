# Feishu 长任务后台执行与定期汇报默认规范（正式草案）

## 1. 目标

本规范用于把项目专项的“Feishu 后台任务提醒与定期汇报机制”提升为通用长任务默认流程，统一解决以下问题：

1. 长任务转入后台后，完成或失败时能够自动提醒。
2. 长任务运行期间，能够主动汇报当前进度、已处理数量和 ETA。
3. 长任务中途中断时，已完成的中间结果能够保留并继续复用，避免“算过了但没落盘”。

本规范同时约束文档层、skill 层和任务执行层，便于在不同项目中复用。

## 2. 适用范围

满足以下任一条件时，应按本规范执行：

- 用户明确要求后台执行、周期性提醒、定期汇报或完成后通知。
- 任务预估总时长不少于 10 分钟。
- 任务包含多阶段、多卡并行、多任务并发或大规模批处理。
- 任务一旦中断，若没有持续落盘会造成明显重算成本。

典型适用对象包括：

- 模型训练、评测、推理批处理
- 多 GPU 并行实验
- 长时间的数据清洗、生成、迁移、索引构建
- 任何需要 Feishu 主动提醒和中间结果持久化的后台任务

## 3. 默认组件角色

通用长任务流程至少包含以下角色：

- `sender`
  负责向指定 Feishu `chat_id` 主动发送文本提醒。
- `completion_notifier`
  负责监听任务终态，并保证完成或失败提醒只发一次。
- `progress_reporter`
  负责读取任务状态并在计划检查点发送进度汇报。
- `status_writer`
  负责在任务执行过程中持续更新机器可读状态。
- `resumable_result_sink`
  负责把中间结果和部分成功结果持续落盘，支持中断恢复。

专项实现可以保留原脚本，但在规范层应使用上述职责名，而不是项目名。

## 4. 标准状态与输出文件

每个长任务至少维护以下文件：

- `status.json`
  当前任务状态主文件。记录任务状态、阶段、进度、ETA、输出路径和最后更新时间。
- `snapshot.json`
  当前阶段或当前窗口的聚合快照。记录关键指标、当前累计结果和摘要信息。
- `rows.jsonl`
  成功处理单元的逐条追加结果。单元可以是 sample、batch、file、record 或其他通用 work unit。
- `failed_rows.jsonl`
  失败处理单元的逐条追加结果。
- `result.json` 或等价正式结果文件
  当前已确认的正式结构化结果。
- `result.md` 或等价摘要文件
  面向阅读的结果摘要。
- `notify.sent` 或其他 sentinel file
  标记终态提醒已发送，避免重复提醒。

推荐 `status.json` 至少包含以下字段：

- `task_name`
- `status`
  可取 `pending`、`running`、`finished`、`failed`、`cancelled`
- `stage`
- `started_at`
- `updated_at`
- `estimated_total_seconds`
- `remaining_seconds`
- `processed_units`
- `total_units`
- `progress_ratio`
- `current_metrics`
- `output_paths`
- `last_error`
- `next_report_checkpoint`

如果任务是多并行子任务，还应额外维护：

- `subtasks`
  每个子任务的状态摘要列表或映射。
- `aggregate_progress`
  总体进度视图。

## 5. 执行流程

### 5.1 启动前

在任务正式启动前，执行器应完成以下动作：

1. 判断任务是否属于长任务。
2. 确定是否需要后台执行。
3. 准备状态目录、日志目录、结果目录和 sentinel file 路径。
4. 生成初始 `status.json`。
5. 预估总时长；若暂时无法预估，则标记为“待预估”，并在 warm-up 结束后补算。
6. 基于 ETA 选择汇报策略：无固定里程碑、1/3、或 1/4。
7. 发送一条“已受理并开始执行”的 Feishu 消息。

### 5.2 运行中

任务运行期间应满足：

- 每个 batch、心跳周期或阶段切换时更新 `status.json`。
- 每个 batch、窗口或阶段末刷新 `snapshot.json`。
- 每个成功单元立刻追加写入 `rows.jsonl`。
- 每个失败单元立刻追加写入 `failed_rows.jsonl`。
- 每次重要阶段完成时刷新正式结果文件。

### 5.3 终态

检测到以下任一情况时，应进入终态通知流程：

- `status == finished`
- `status == failed`
- `status == cancelled`
- 主进程退出，且正式输出已生成

进入终态后：

1. 发送一次终态提醒。
2. 写入 sentinel file。
3. 停止 completion watcher 和 progress watcher。

## 6. ETA 预估与汇报策略

### 6.1 初始 ETA

长任务默认要求在启动前或启动后短暂 warm-up 阶段内给出初始 ETA。

可接受的 ETA 来源包括：

- 历史同类任务平均耗时
- 已知样本量与单样本耗时估算
- 已知 batch 数与单 batch 耗时估算
- 已知阶段模板的经验耗时

如果启动时无法可靠预估，允许先发送“ETA 待 warm-up 后补算”，但应在尽快具备样本后补充初始 ETA。

### 6.2 默认分段规则

固定的 `5400` 秒或“每 1.5 小时一次”不再作为默认规范。

统一改为基于预估总时长的里程碑汇报：

- ETA 小于 45 分钟
  不强制固定里程碑；发送启动通知、关键事件通知和终态通知即可。
- ETA 大于等于 45 分钟且小于 3 小时
  按 1/3 里程碑汇报，默认检查点为约 33% 和 67%。
- ETA 大于等于 3 小时
  按 1/4 里程碑汇报，默认检查点为约 25%、50%、75%。

这里的“1/3”与“1/4”默认指按预估总时长分段，而不是按固定秒数分段。

### 6.3 重估规则

当以下任一情况出现时，应重估 ETA，并重排尚未发送的后续汇报点：

- 实际速度相对初始估计偏差不小于 25%
- 绝对剩余时长偏差不小于 20 分钟
- 阶段切换导致耗时模型明显变化
- 并行子任务数量变化或有任务失败重启
- 任务在一个或多个轮询窗口内明显停滞

重估后应遵循：

- 已发送过的里程碑不补发
- 仅重排尚未发送的后续检查点
- 下一次汇报中明确说明 ETA 已更新

## 7. 汇报触发条件

### 7.1 计划性汇报

计划性汇报由里程碑检查点触发，默认仅在达到未发送的 1/3 或 1/4 检查点时发送。

### 7.2 立即汇报

即使尚未达到计划检查点，以下情况也应立即汇报：

- 任务刚启动并进入后台
- 初始 ETA 首次可用
- 阶段发生重大切换
- 首个可用中间结果生成
- 并行子任务出现失败、重启或明显拖慢总体进度
- 任务被取消
- 任务完成或失败

## 8. 汇报内容规范

每条运行中进度汇报至少应包含：

- 任务名
- 当前阶段
- 已处理数量 / 总量
- 当前进度百分比
- 当前 ETA 或剩余时间
- 当前主要输出目录或关键中间结果路径
- 下一次预计汇报点

多任务并行时，还应包含：

- 已完成子任务数 / 总子任务数
- 总体聚合进度
- 总体 ETA
- 当前最慢或阻塞的子任务

如果任务是按样本处理，推荐明确写出“已处理样本数”。
如果任务不是样本型任务，统一使用“已处理单元数”。

## 9. 完成与失败提醒规范

完成或失败提醒应满足：

- 只发送一次
- 能区分 `finished`、`failed`、`cancelled`
- 能附带关键输出位置
- 能附带最终摘要或主要指标

完成提醒建议包含：

- 任务名
- 终态
- 总耗时
- 输出文件路径
- 如有必要，附上正式结果摘要

失败提醒建议包含：

- 任务名
- 失败阶段
- 最近错误摘要
- 已保留中间结果的位置
- 是否可恢复、从哪里恢复

## 10. 持久化与可恢复原则

长任务默认遵循“先落盘，再继续”的原则。

最低要求如下：

- 每个 batch 或处理窗口更新 `status.json`
- 每个 batch 或处理窗口更新 `snapshot.json`
- 每个成功单元立刻落入 `rows.jsonl`
- 每个失败单元立刻落入 `failed_rows.jsonl`
- 每个阶段完成后刷新正式结果文件

推荐的恢复能力包括：

- 根据 `rows.jsonl` 或索引清单跳过已完成单元
- 根据 `failed_rows.jsonl` 单独补跑失败单元
- 根据 `snapshot.json` 恢复阶段级统计
- 重启后继承之前的 `status.json` 和结果目录，而不是重头开始

## 11. 多并行子任务聚合规则

对于多 GPU 或多进程并行任务，建议使用以下聚合约定：

- 每个子任务维护独立 `status.json`
- 聚合器周期性读取全部子任务状态
- 总体已处理数量为各子任务 `processed_units` 之和
- 总体总量为各子任务 `total_units` 之和
- 全并行任务的总体 ETA 默认取活动子任务 `remaining_seconds` 的最大值
- 串并混合任务可取“当前活动最长分支 + 尚未启动阶段估计值”的组合

## 12. 与项目专项实现的映射

以 `weather_qwen` 为例，可做如下抽象：

- `send_feishu_text.py`
  抽象为 `sender`
- `task_completion_notifier.py`
  抽象为 `completion_notifier`
- `parallel_progress_reporter.py`
  抽象为 `progress_reporter`
- `evaluate_weather_qwen_compare.py` 中的状态更新逻辑
  抽象为 `status_writer` 与 `resumable_result_sink`
- `processed_samples`
  抽象为 `processed_units`
- `5400` 秒定时汇报
  替换为 `estimated_total_seconds + reporting_fraction`

这意味着：

- 项目可以保留原脚本，不必一次性重写
- 但对外规范、skill、通用接口和新项目复用时，应使用通用命名和默认策略

## 13. v1 边界

本规范 v1 默认接受以下边界：

- 进度汇报仍可基于“轮询状态文件”，不强制要求事件驱动
- 详细进度以 `status.json` 与 `snapshot.json` 为准，日志仅作辅助
- 恢复能力可以先做到“保留并可人工续跑”，再逐步升级到“自动跳过已完成单元”

## 14. 推荐的 skill 草案结构

推荐 skill 采用“简短 SKILL.md + 详细 references”的结构：

- `SKILL.md`
  仅保留触发说明、默认流程和关键约束
- `references/formal-spec.md`
  保存本正式规范
- 后续如需实现模板，可再补：
  - `references/status-schema.md`
  - `references/example-layout.md`
  - `scripts/` 中的 sender / notifier / reporter 模板

## 15. 结论

从本规范起，Feishu 长任务默认流程应从“固定时间轮询汇报”升级为：

- 先估总时长
- 再选择 1/3 或 1/4 里程碑
- 运行中按里程碑和关键事件主动汇报
- 中间结果持续落盘
- 完成或失败时只提醒一次

该规则既兼容现有专项实现，也适合作为后续通用 skill 的默认执行协议。
