# Public / Private 边界说明

## 1. 这份边界说明解决什么问题

你当前的目标不是把所有东西都放进一个仓库，而是：

- 框架和公开 skill 保持 public
- 科研任务代码和敏感层保持 private
- 运行态永远留在本机

## 2. 应该进 public 的内容

适合进 public：

- `gateway/` 的通用源码
- 启动脚本、计划任务脚本、健康检查脚本
- 脱敏配置模板
- 公开可复用的 skills
- 架构说明
- 部署文档
- 通用导出和报送脚本
- 可公开的测试

## 3. 应该进 private 的内容

适合进 private：

- 科研代码
- 实验脚本
- 私有 skill
- 内部 prompt
- 任务相关数据接口
- 只针对某个课题组或项目的工作流
- 不便公开的模型调用约定

## 4. 不应该进任何仓库的内容

这些默认只放本机：

- `appId`
- `appSecret`
- 真实 chat ID
- 真实 open ID
- 本机运行日志
- 中间附件
- SQLite 状态库
- 运行中的 `status.json`
- `.codex-feishu-gateway/`
- `.codex-long-tasks/`

## 5. 对 public 仓库最重要的一条规则

只提交“别人拿去也能复用”的东西。

如果某个文件离开你的机器就失去意义，或者包含真实环境细节，那它大概率不应该进 public。

## 6. 对 private 仓库最重要的一条规则

只把任务层和敏感层放进去，不要再复制一份 public 框架。

private 仓库应该建立在 public 框架之上，而不是重新造一个平行框架。

## 7. 推荐的落地方式

推荐形态：

- 一个 public 框架仓库
- 一个 private 科研仓库
- 多台机器上的本地运行态目录

这样后面让 Agent 自动同步时，边界最清晰，也最不容易把敏感内容误提交到 public。
