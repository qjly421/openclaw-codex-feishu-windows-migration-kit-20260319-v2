# Codex Feishu Framework 范式

## 一、定位

这份文档是 GitHub 中的详细版范式说明。

目标不是描述单个 skill，而是定义一套可复用的工作方式，让后来的 AI 和新人都能沿用已经验证过的路径，减少重复对话、重复排查和重复试错。

飞书中的同名页面只保留人类入口版；这里保留详细版。

## 二、核心结构

这套范式只分两层。

### 1. 沉淀层

沉淀层负责“存什么”。

包括：

- 历史案例
- 信源与数据组织
- 密钥位置说明
- 给人的交付模板

### 2. 执行层

执行层负责“怎么用”。

包括：

- 步骤
- 权限停点
- agent 分工
- 什么时候复用旧路径
- 什么时候必须重查或重跑

## 三、平台分工

### 1. GitHub

GitHub 是主沉淀库。

放这些内容：

- `gateway/`
- `skills/`
- `docs/`
- `templates/`
- 脱敏后的历史案例
- 可复用脚本和结构化规范

GitHub 的职责是让 AI 和维护者可以稳定复用。

### 2. 飞书

飞书是人类入口。

放这些内容：

- 人类可读的总入口
- 新人接手说明
- 简短的复用规则
- 内部说明
- 不适合放到 GitHub 的敏感边界说明

飞书的职责是让人愿意用、会用、知道去哪找。

### 3. 本机

本机只作为运行时工作区。

放这些内容：

- 缓存
- 临时文件
- 运行中间产物
- 本地配置副本

本机不是主沉淀库。

### 4. 密钥

密钥独立管理。

共享材料中只允许记录：

- 密钥名称
- 用途
- 所有人
- 存放位置
- 获取方式

不允许记录：

- live token
- app secret 明文
- 私钥正文

## 四、当前第一版范式范围

当前第一版只覆盖这几个方向：

1. GitHub 接入与同步
2. Feishu wiki/docx 读写
3. public skill 复用
4. 多设备部署与迁移参考

先把这一版打稳，再扩别的范式。

## 五、默认执行规则

所有 agent 默认按下面顺序做事：

1. 先读相关沉淀
2. 再做本机检查
3. 再验权限
4. 再做写入
5. 写完回读验证
6. 补充可复用记录

### 对 GitHub

至少先检查：

- `gh auth status`
- repo 当前 branch
- remote
- working tree 是否干净
- 目标 repo / branch 是否允许写

### 对 Feishu

至少先检查：

- token 是否可申请
- 目标 wiki 节点是否可读
- 目标父节点是否可创建子页
- 目标 docx 是否可写

### 写入后

必须做回读校验，确认实际结果和预期一致。

## 六、复用旧路径的条件

满足下面条件时，可以优先复用旧路径：

- 同一个 GitHub repo
- 同一个 Feishu 空间或同类目标
- 相同认证方式
- 相同写入边界
- 上一次同类执行成功

## 七、必须重查或重跑的条件

遇到下面情况，不要直接复用旧路径：

- 换机器
- 换账号
- 换 branch
- 换 wiki 节点或 parent node
- 换 secret 来源
- 上一次执行失败
- 目标权限模型发生变化

## 八、对 agent 的统一要求

任何按这套范式工作的 agent，都默认遵守这些要求：

1. 先读 GitHub 主仓库里的 README 和相关 docs
2. 不跳过只读检查
3. 不把 live secret 写进共享文档或 public repo
4. 对写入动作显式做前置检查和后置回读
5. 如果边界变化，不硬套旧步骤

## 九、与 skill 的关系

skill 不是这套范式本身。

更准确地说：

- 范式是上层规则
- skill 是入口和执行包装

所以：

- GitHub 里保留详细规则
- skill 负责让 AI 快速进入对应工作流
- 飞书负责让人快速理解和接手

## 十、与 public/private 分层的关系

这套范式默认：

### public repo

放：

- 公共安全的代码
- 公共 skill
- 脱敏文档
- 结构化规范
- 公共模板

### private repo

放：

- 私有科研代码
- 敏感配置
- 私有流程
- 实验输出
- 内部任务逻辑

## 十一、当前推荐做法

当前最小可用做法就是：

1. GitHub 继续作为主沉淀库
2. 飞书只做人类入口
3. 本机只做运行时承接
4. 密钥独立管理
5. 先只维护 GitHub + Feishu 这一条范式样板

## 十二、给新人的最短入口

新人接手时：

1. 先看 `README.md`
2. 再看本文件
3. 再看 `docs/PUBLIC-PRIVATE-BOUNDARY-ZH.md`
4. 根据任务类型进入对应的 skill 或脚本

如果不确定是否能复用旧路径，默认先重查，不要直接写。
